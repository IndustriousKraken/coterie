# totp-2fa Specification Delta

## MODIFIED Requirements

### Requirement: Two-step enrollment with no DB write before confirmation

Enrollment SHALL be:

1. `begin_enrollment(account_name)` — generates a fresh 20-byte secret in memory and returns the otpauth:// URL, base32 secret, and an SVG QR code. **No database write.**
2. `confirm_enrollment(member_id, secret_base32, code, account_name)` — verifies the supplied code matches the secret, then writes encrypted secret + `totp_enabled_at` to the `members` row, **but ONLY when the member is not already enrolled**. If `totp_enabled_at` is already set, `confirm_enrollment` SHALL return `Ok(false)` without touching the row, without decoding the supplied secret, and without verifying the supplied code. Rotation goes through the `disable` flow (which requires a current code) and then a fresh enrollment.

The plaintext secret SHALL round-trip through the enrollment page in a hidden field; this is the only path it takes between begin and confirm.

#### Scenario: Abandoned enrollment leaves no DB trace

- **WHEN** a member starts enrollment but never submits the confirmation code
- **THEN** the database SHALL contain no record of the started enrollment

#### Scenario: Confirmation with valid code persists encrypted secret

- **WHEN** `confirm_enrollment` receives a code that verifies against the supplied secret AND the member is not already enrolled
- **THEN** the row's `totp_secret_encrypted` and `totp_enabled_at` columns SHALL be set in a single UPDATE; recovery-code generation SHALL be a separate caller-driven step

#### Scenario: confirm_enrollment refuses to overwrite an existing secret

- **WHEN** `confirm_enrollment` is called for a member whose `totp_enabled_at` is already non-null
- **THEN** the call SHALL return `Ok(false)`, the row's `totp_secret_encrypted` and `totp_enabled_at` SHALL be unchanged, and the recovery-codes column SHALL NOT be regenerated by the caller's "enrollment confirmed" branch
- **AND** an authenticated-session attacker who POSTs an attacker-controlled `secret_base32` plus a matching code SHALL NOT be able to silently rotate the victim's TOTP secret without first calling `disable` (which itself requires a current code)
